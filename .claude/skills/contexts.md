# Project Context Map

This project has a committed **`context-map.json`** at the repo root, generated by Vibeman. It groups every source file into a named **context** (a feature) under a business **domain**. It is the source-of-truth map of "what lives where" — and it works with **zero dependency on Vibeman running**. Treat it as always-present orientation data.

## When to use

- **At the start of any non-trivial coding task** — to scope your work to the right files.
- **After you add/move/delete files for a feature** — to keep the map accurate for the next session.

## 1. Orient (read first)

Read `context-map.json`. The fast path is the top-level `index` array — one line per context (`group › context [category] — N files`). Then:

1. Find the context whose `businessFeature`/`name`/`description` matches your task.
2. Read its `filePaths` — those are the files to focus on. Read its group's siblings + the `relationships` array to understand what it depends on.
3. Scope your change to that context's files. If your change clearly spans two contexts, that's a signal they may be coupled — note it.

```bash
# Quick peek at the index
cat context-map.json | node -e "const m=JSON.parse(require('fs').readFileSync(0,'utf8'));(m.index||[]).forEach(l=>console.log(l))"
```

## 2. Maintain (after you change files)

When you add, move, or remove files for a feature, keep the map honest:

- **Vibeman is running** (`http://localhost:3000` reachable)? Prefer the live tools/API so the DB and file stay in sync:
  - MCP: `update_context` / `create_context` / `refresh_context`, or `audit_contexts` to check balance.
  - Or `curl -X PUT http://localhost:3000/api/contexts -H 'Content-Type: application/json' -d '{"contextId":"...","updates":{"file_paths":[...]}}'`
  - Vibeman re-writes `context-map.json` automatically after the change.
- **Vibeman is NOT running**? Edit `context-map.json` directly: add/remove the path in the relevant context's `filePaths` (keep each context to ~5–15 files, every file in exactly one context). Leave a note that the map was edited offline. When Vibeman is next online, run MCP `sync_context_map` (or `curl -X POST http://localhost:3000/api/contexts/import -d '{"projectId":"<id>"}'`) to reconcile your edits back into Vibeman.

## Rules of thumb (match Vibeman's policy)

- A context is a **user capability / feature**, not a layer. 5–15 files (split if over 15, merge if under 3 sharing a table/API).
- Every context has a `category` (`ui|api|lib|data|test|config`); every group a `domain` (`feature|infrastructure|shared|integration|data`).
- Every meaningful source file belongs to **exactly one** context — no overlaps, no gaps.

## Do NOT

- Don't delete contexts/groups by removing them from the file expecting a sync to delete them — `sync_context_map` is **upsert-only** (safe by design). Use the Vibeman UI/API to delete.
- Don't let a context balloon past ~15 files — split it.
