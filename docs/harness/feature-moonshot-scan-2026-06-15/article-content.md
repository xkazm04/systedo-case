## 1. Promote the single article into a typed content collection (slug routing + multi-article)
- **Severity**: High
- **Lens**: feature-scout
- **Category**: functionality
- **Effort**: M (1-3d)
- **File**: src/lib/article.ts (`import data from "@/data/article.json"`, `export const article`) + src/app/clanek/page.tsx + src/lib/nav.ts (`categoryHubPath`, `slugify`)
- **Scenario**: The model is genuinely headless, but the loader hardcodes one file (`import data from "@/data/article.json"`) and the route is the static `/clanek`. `nav.ts` already openly anticipates the next step — `categoryHubPath()` builds `/clanek?kategorie=…` and its comment says "once it grows into a listing hub the same URL filters by category." The infrastructure is half-built but wired to exactly one article.
- **Opportunity**: Turn `src/data/article.json` into `src/data/articles/*.json`, add an `Article["meta"].slug` field, and replace the single import with a loader: `getAllArticles()`, `getArticleBySlug(slug)`, `getArticlesByCategory(cat)`. Convert `/clanek` into `/clanek/[slug]` (static `generateStaticParams` from the collection) plus a `/clanek` listing hub that reads `categoryHubPath`'s `?kategorie=` filter. All derivation helpers (`tableOfContents`, `figureBlocks`, `inlineToText`) already take an `Article` argument, so they need zero changes — only the JSON-LD/metadata block in `page.tsx` moves from module scope into the route component.
- **Impact**: Unlocks the entire "blog" narrative the case study gestures at, makes the headless-CMS claim real (a CMS with one document is a hardcoded page), and turns the `categoryHubPath`/`slugify` dead-ends into live features. It is the prerequisite for ideas #2, #4 and #5.
- **Implementation sketch**: (1) Add `slug: string` to `Article["meta"]`. (2) In `article.ts`, replace the static import with `import.meta.glob`-style eager JSON import (or a small `fs`-free static map indexed by slug) and export `getAllArticles`/`getArticleBySlug`. (3) Create `src/app/clanek/[slug]/page.tsx` from the current `page.tsx`, moving `jsonLd`/`breadcrumbs`/`metadata` into `generateMetadata`/the component keyed on the resolved article. (4) Add `generateStaticParams` returning every slug. (5) Build a `/clanek/page.tsx` index that lists articles and honours `searchParams.kategorie`.

## 2. Build-time content validator for the Article JSON (links, anchors, figures, schema)
- **Severity**: High
- **Lens**: feature-scout
- **Category**: automation
- **Effort**: M (1-3d)
- **File**: src/lib/article.ts (`Inline`, `Block`, `tableOfContents`, `figureBlocks`) — new `src/lib/article-validate.ts`
- **Scenario**: The content model has rich invariants the type system can't catch: anchor links (`kind: "anchor"`, e.g. `#kvalita`) must point at a real `HeadingBlock.id`; figure `src` paths (`/clanek/prehled-druhu.svg`) must exist on disk; every figure needs non-empty `alt` (it feeds both accessibility and the `ImageObject.description` in JSON-LD); heading `id`s must be unique (the TOC and anchors depend on it). Today an author editing raw JSON gets a green build and a silently broken anchor or a 404 image with no warning.
- **Opportunity**: A pure `validateArticle(a: Article): Issue[]` function plus a tiny script (`scripts/check-content.ts`) run in CI / `prebuild`. Checks: (a) every `kind:"anchor"` href resolves to an existing heading id; (b) every `FigureBlock.src` file exists under `public/`; (c) `alt` non-empty and `width`/`height` present (CLS + rich-results readiness); (d) duplicate heading ids; (e) external links are absolute `https://`; (f) FAQ answers non-empty after `inlineToText`. Reuse `tableOfContents`/`figureBlocks` to gather targets so the validator and the renderer share one source of truth.
- **Impact**: Makes the headless model authorable by hand without footguns, protects the SEO payload (broken anchors/missing alt quietly degrade the JSON-LD this context is proud of), and demonstrates engineering maturity that suits a job-application case study. Scales directly with idea #1 (validate the whole collection).
- **Implementation sketch**: (1) New `src/lib/article-validate.ts` exporting `validateArticle`. (2) Collect heading ids via `tableOfContents` + walk `h3` blocks; collect anchors by scanning `Inline[]` for `kind:"anchor"`. (3) For figures, `existsSync(path.join("public", f.src))`. (4) `scripts/check-content.ts` imports the article(s), prints issues, `process.exit(1)` on any error. (5) Add `"prebuild": "tsx scripts/check-content.ts"` to package.json scripts.

## 3. In-repo authoring & preview studio for the block model (`/design-system`-style)
- **Severity**: Medium
- **Lens**: feature-scout
- **Category**: user_benefit
- **Effort**: M (1-3d)
- **File**: src/components/article/ArticleBody.tsx (the `switch (block.type)` renderer) + new `src/app/studio/page.tsx` (mirrors existing src/app/design-system/page.tsx)
- **Scenario**: `ArticleBody.tsx` is already a complete, isolated block renderer that maps every `Block` variant to JSX, and the repo already ships a `/design-system` gallery page — so the project clearly values visual component catalogs. But there is no way to see all block variants at once, or to paste candidate Article JSON and preview it before committing. Authors hand-edit `article.json` blind.
- **Opportunity**: A `/studio` route with two panes: left, a textarea bound to live `JSON.parse` of pasted Article JSON; right, the real `<ArticleBody>` + derived `<ArticleToc>` rendered from it, plus a live `validateArticle` (idea #2) issue list and a preview of the generated JSON-LD graph. Below, a static "block gallery" rendering one of every `Block` variant (h2/h3/p/ul/ol/callout/quote/cta/stat/figure) as living documentation of the content model.
- **Impact**: Turns "headless CMS" from a data shape into an actual authoring surface, gives reviewers a tangible way to explore the model, and is the human-in-the-loop editor that the AI pipeline (idea #5) writes into. Low risk — it reuses the existing renderer verbatim.
- **Implementation sketch**: (1) Copy the structure of `src/app/design-system/page.tsx` into `src/app/studio/page.tsx` (client component). (2) `const [raw, setRaw] = useState(...)`; `useMemo` to `JSON.parse` with a try/catch surfacing parse errors. (3) Render `<ArticleBody blocks={parsed.blocks} />`, `<ArticleToc items={tableOfContents(parsed)} />`, and a `<pre>` of the JSON-LD built by extracting the graph-builder from `clanek/page.tsx` into a shared `buildArticleJsonLd(a)` helper in `article.ts`.

## 4. AI pipeline that generates a complete structured Article JSON from a topic (figures + FAQ + JSON-LD ready)
- **Severity**: Critical
- **Lens**: moonshot-architect
- **Category**: automation
- **Effort**: L (>3d)
- **File**: src/lib/gemini.ts (the tool pattern, esp. `generateBrief`/`BRIEF_SCHEMA`) + src/lib/llm/index.ts (`generateStructured`) + src/lib/article.ts (`Block`, `Article` schema) + new `src/app/api/ai` mode `"article"`
- **Scenario**: Two halves of this pipeline already exist and have never been connected. `generateBrief` already produces an `outline` (H2 + bullet points), a `faq` array, a `slug`, `titleTag`/`metaDescription` and `internalLinks` — i.e. the *skeleton* of an article — but it stops at a brief. Separately, this context defines the exact typed `Block`/`Inline`/`FaqItem` model the site renders. Nothing turns one into the other. The moonshot is to close the loop: topic → full, renderable, schema-valid Article JSON.
- **Opportunity**: Add a fifth tool, `generateArticle(req)`, that emits the *entire* `Article` shape — `meta`, `blocks[]` (typed h2/h3/p/ul/callout/stat/figure/cta with first-class `Inline` links + anchors), and `faq[]` — as Gemini/Claude structured output. Encode the `Block` union as a discriminated `Type.OBJECT` schema, run it through the existing `generateStructured` chokepoint (free demo fallback + provider switch already handled), then pipe the result through `validateArticle` (idea #2) and write it into the collection (idea #1). For figures, have the model emit `FigureBlock` nodes with `alt`/`caption`/intrinsic dimensions and a placeholder `src`, so the JSON-LD `ImageObject` graph populates automatically. Optionally chain `generateBrief` → `generateArticle` so the brief becomes the planning step.
- **Impact**: This is the category-defining move for a *marketing*-analytics case study: a one-prompt "write me a publish-ready, SEO-structured, schema.org-rich article" generator that lands directly in the headless model the site renders — no copy-paste, no hand JSON. It makes every other idea here a force multiplier (studio = its editor, validator = its guardrail, collection = its output store) and is a genuinely impressive demo for the product's content pillar (Tvorba obsahu).
- **Implementation sketch**: (1) In `ai-types.ts` add `ArticleRequest`/`validateArticleRequest` (topic, primaryKeyword, audience, optional sourceBriefSlug). (2) In `gemini.ts`, define `ARTICLE_SCHEMA` mirroring the `Block` union (use a `type` enum + variant-specific optional props, since Gemini schemas can't truly union — normalize defensively in `normalizeArticle`, exactly as `normalizeBriefResult` already does). (3) `ARTICLE_SYSTEM` prompt reusing the Czech editorial rules from `BRIEF_SYSTEM` plus "emit anchor links to your own headings, 2 figures with descriptive alt, a stat block, one CTA." (4) Add `case "article"` to `src/app/api/ai/route.ts`. (5) On success, `validateArticle` then persist into the collection; surface in the `/studio` editor for human review before commit.

## 5. Render the dashboard's data snapshot AS an auto-generated, embeddable data-story article
- **Severity**: High
- **Lens**: moonshot-architect
- **Category**: integration
- **Effort**: L (>3d)
- **File**: src/lib/snapshot.ts (`buildSnapshot`, `snapshotToPromptText`) + src/lib/gemini.ts (`generateAnalysis`, `ANALYSIS_SCHEMA`) + src/lib/article.ts (`StatBlock`, `FigureBlock`, `Block`)
- **Scenario**: The performance-analysis tool already turns the real dashboard dataset into a structured narrative (`headline`, `summary`, `wins`, `risks`, `actions`) grounded strictly in `buildSnapshot(period)`. Meanwhile this content context has a `StatBlock` (value/label pairs — perfect for KPIs) and a `FigureBlock` (chart image). Today the analysis renders only as ephemeral cards on `/ai-asistent`; it is never crystallised into a durable, shareable, SEO-indexed artifact. The data and the narrative format are both here; the bridge isn't.
- **Opportunity**: A `snapshotToArticle(snapshot, analysis): Article` adapter that composes the analysis into the typed block model: the headline → `meta.title` + perex; the snapshot's headline KPIs → a `StatBlock`; `wins`/`risks` → `ul` blocks under `h2` headings (with auto-generated anchor ids feeding `tableOfContents`); `actions` → an `ol`; a rendered dashboard chart → a `FigureBlock` promoted into the JSON-LD `ImageObject` graph; and the analysis questions → `faq[]`. The result is a real article object the existing `/clanek/[slug]` route (idea #1) renders and Google can index as a `FAQPage` + `Article` — a "monthly performance report as a public data story," regenerated whenever the data updates.
- **Impact**: Closes the loop between the analytics pillar and the content pillar — the two halves of the case study become one product: data in → published, structured, schema-rich narrative out. It is a force multiplier with network-effect potential (every period/segment becomes an evergreen indexed page), and it showcases the headless model doing what hand-authored content can't: scaling from live data with zero manual JSON.
- **Implementation sketch**: (1) New `src/lib/snapshot-to-article.ts` exporting `snapshotToArticle`. (2) Map `Snapshot.current` KPIs → `StatBlock.items`, `AnalysisResult.wins/risks/actions` → list/ol blocks, generating slugged heading ids (reuse `gemini.ts`'s `slugify`). (3) Reference a chart image (static SVG, or a future server-rendered chart) as a `FigureBlock` with intrinsic `width`/`height`. (4) Run through `validateArticle` (idea #2) and the shared `buildArticleJsonLd` helper. (5) Expose at `/clanek/report-<period>` via `generateStaticParams`, or as a "Publikovat jako článek" action in the analysis UI.
