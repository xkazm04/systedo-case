## 1. Stop hand-typing the project-type domain — import `PROJECT_TYPES` instead of re-declaring it

- **Severity**: High
- **Category**: duplication
- **File**: `src/lib/ai/tools/onboarding-scan.ts:40-46,107`
- **Scenario**: `onboarding-scan.ts` declares its own `TYPE_HINT: Record<string, string>` (lines 40-46) and its own `KNOWN_TYPES = new Set(["eshop", "app", "leadgen", "content", "local"])` (line 107). `channel-research.ts:56-64` declares the *same* five-key Czech description map (`TYPE_FRAMING`) — byte-for-byte identical text to `TYPE_HINT` ("e-shop (prodej fyzického zboží)", "digitální produkt / SaaS aplikace", …) — plus its own `KNOWN_TYPES` Set with the same five literals. Both are redundant with `PROJECT_TYPES = ["eshop", "app", "leadgen", "content", "local"]`, already exported from `@/lib/projects/types.ts:17`, which `channel-research.ts:22` already imports the `ProjectType` *type* from. The same "is this string a known project type" check is independently re-implemented a third and fourth time app-wide in `src/app/api/projects/route.ts:17-18` and `src/app/api/projects/[id]/route.ts:12-13` (`isProjectType`), and `src/lib/ai/validation.ts:787` already builds `PROJECT_TYPE_SET = new Set<string>(PROJECT_TYPES)` and uses it to validate `req.projectType` on **both** `ChannelResearchRequest` and `OnboardingScanRequest` *before* they ever reach these two tool files — so `resolveType`/`KNOWN_TYPES` here are re-validating something the caller already guarantees.
- **Root cause**: Each tool file was written to be self-contained (own prompt, own coercion), so nobody reached for the shared `@/lib/projects/types.ts` module already open one import away.
- **Impact**: A sixth `ProjectType` would need updating in five places to stay correct app-wide. `channel-research.ts`'s `TYPE_FRAMING` is typed `Record<ProjectType, string>`, so TypeScript *would* catch a missing key there — but `onboarding-scan.ts`'s `TYPE_HINT` is typed `Record<string, string>`, so a new type would silently get no hint (a degraded prompt, no compile error). That's the concrete bug-cost: one of the two copies has no type-level safety net at all.
- **Fix sketch**: In `onboarding-scan.ts`, `import { PROJECT_TYPES, type ProjectType } from "@/lib/projects/types"` and replace `KNOWN_TYPES` with `new Set<string>(PROJECT_TYPES)` (or reuse a shared `isProjectType` guard hoisted into `@/lib/projects/types.ts` next to `PROJECT_TYPES`, which would also let the two `api/projects/*` routes drop their own copies — out of scope here but worth a follow-up). Re-key `TYPE_HINT`/`TYPE_FRAMING` as `Record<ProjectType, string>` in both files (or hoist the one shared Czech description map into a small shared module) so a future type addition is a compile error in both places, not just one.
- **Gate impact**: Touching `channel-research.ts` and `onboarding-scan.ts` re-proves only the `channel-research` and `onboarding-scan` tools (each ~25s, per-file attribution in `scripts/llm-gate.mjs`) — **not** a full gate run, since `@/lib/projects/types.ts` itself is not in `HASHED_FILES`.

## 2. Four hand-rolled "coerce to known enum" helpers, same three-line shape

- **Severity**: Medium
- **Category**: duplication
- **File**: `src/lib/ai/tools/channel-research.ts:153-161`
- **Scenario**: `coerceCategory` (153-156) and `coerceEffort` (158-161) in this file, and `coerceCause` (`lead-source-diagnosis.ts:120-123`) and `coerceSeverity` (`lead-source-diagnosis.ts:125-128`) are the identical shape four times over: lowercase the trimmed text via `txt(v).toLowerCase()`, check membership in a `Set<string>` built from an exported const-array type, and cast to the enum type (falling back to a literal default, or `undefined` for severity). Each is 3-4 lines; together they're ~14 lines of boilerplate that says the same thing four times across two files.
- **Root cause**: Every tool that needs "the model returned a free-text field but it must be one of N known values" wrote its own micro-coercer instead of reaching for a shared one — there isn't one to reach for yet.
- **Impact**: Low bug risk today (every instance degrades safely to a default), but it's the kind of boilerplate that gets copy-pasted a fifth and sixth time into new tools (e.g. `keyword-clusters.ts`'s inline `INTENTS.has(intent) ? intent as … : undefined` at line 229 is already a fifth near-instance) instead of being written once.
- **Fix sketch**: Add one generic helper to `_shared.ts`, e.g. `coerceEnum<T extends string>(v: unknown, set: ReadonlySet<string>, fallback?: T): T | undefined` (lowercased, trimmed via the existing `txt`), and replace all four call sites (plus the `keyword-clusters.ts` inline check) with it.
- **Gate impact**: `_shared.ts` **is** in `scripts/llm-gate.mjs`'s `HASHED_FILES` (line 45) but carries no `// llm-tool:` tags of its own, so per the gate's own attribution logic ("a change with no per-tool delta … conservatively re-runs everything") editing it forces a **full real-model gate run across all 19 tools**, not just the four touched here. That's the real cost of this consolidation — worth it once, but call it out before doing it (and batch it with any other `_shared.ts` change, e.g. finding 4, into the same commit rather than paying the full-suite cost twice).

## 3. `twin-reply.ts` and `twin-style.ts` each carry their own Czech label map for the same `TwinChannel` domain

- **Severity**: Medium
- **Category**: duplication
- **File**: `src/lib/ai/tools/twin-reply.ts:27-35`
- **Scenario**: `TWIN_CHANNEL_LABELS` here maps every `TwinChannel` (`leads/email/chat/social/reviews/sms/whatsapp`, from `lib/twin/types.ts`) to a Czech phrase used in "Napiš další odchozí zprávu na kanálu: …". `twin-style.ts:23-32`'s `SCOPE_LABELS` maps the same seven keys (plus `generic`) to near-identical Czech phrases used in "Vytáhni komunikační styl … pro kanál: …" — five of the seven values differ only in grammatical number/register (`"poptávka"` vs `"poptávky"`, `"veřejná recenze"` vs `"veřejné recenze"`, `"sociální síť (komentář / DM)"` vs `"sociální sítě"`); two (`email`, `chat`) are byte-identical. Neither is exported or reused elsewhere, so this is two independent hand-typed copies of one lookup table.
- **Root cause**: Both tools needed "a human-readable name for this channel, for this sentence" and each just wrote its own map rather than sharing the enum-to-label mapping and letting the sentence supply its own grammatical framing.
- **Impact**: Both maps are typed `Record<string, string>` rather than `Record<TwinChannel, string>` / `Record<ToneScope, string>`, so TypeScript does not enforce exhaustiveness against `TWIN_CHANNELS`/`TONE_SCOPES` in `lib/twin/types.ts`. If a channel is ever added there, neither map gets a compile error — the code silently falls back to the raw key (`TWIN_CHANNEL_LABELS[req.channel] ?? req.channel`) in one file's output and the raw scope key in the other's, degrading the prompt without any signal.
- **Fix sketch**: Re-key both as `Record<TwinChannel, string>` / `Record<ToneScope, string>` (import the types from `lib/twin/types.ts`) so a new channel is a compile error in both files immediately. If it's worth removing the duplication itself (not just making it safe), hoist one canonical `Record<TwinChannel, string>` of base nouns and derive each sentence's grammatical form from it in a small helper — but the two phrasings are genuinely different registers (singular "this one reply" vs. plural "this channel in general"), so verify with the copy owner before merging content, not just types.
- **Gate impact**: Re-typing in place touches only `twin-reply.ts` and `twin-style.ts`, both individually tagged/hashed — re-proves just the `twin-reply` and `twin-style` tools, not a full run. A shared-helper extraction into `_shared.ts` (or the already-hashed `voice.ts`) would pay the full-suite cost described in finding 2; a new *un-hashed* module under `lib/twin/` would not.

## 4. Two independent 0-100 clamp implementations

- **Severity**: Medium
- **Category**: duplication
- **File**: `src/lib/ai/tools/channel-research.ts:163-167`
- **Scenario**: `clampFit` here (`Math.round(Number(v))`, NaN→60, then `Math.max(0, Math.min(100, n))`) and `twin-reply.ts:131-134`'s `clampScore` (`typeof v === "number" ? v : 0`, then the same `Math.max(0, Math.min(100, Math.round(n)))`) do the same round-and-clamp-to-0-100 arithmetic with only the invalid-input fallback (60 vs 0) differing. A third sibling, `src/lib/ai/tools/campaign-eval.ts:82`, does it again inline (`Math.max(0, Math.min(100, Math.round(raw)))`) — not owned by this context, cited only as evidence this shape keeps getting re-typed rather than shared. `lib/twin/types.ts:265-268` also has its own private, fully generic `clamp(v, lo, hi, fallback)` used to re-clamp `TwinDraft.confidence` on the way into storage — meaning the value `twin-reply.ts`'s `clampScore` produces gets clamped a second time, by a second implementation, one file downstream.
- **Root cause**: A one-off numeric clamp is cheap enough to write inline each time, so nobody centralized it even though `_shared.ts` already centralizes the equivalent string-clamp (`clamp(s, n)`).
- **Impact**: Low — both current instances are correct and cheap. The cost is purely "one more thing a future tool author reinvents" rather than any live bug.
- **Fix sketch**: Add `clampScore(v: unknown, fallback = 0): number` to `_shared.ts` (name it distinctly from the existing string `clamp` to avoid confusion) and use it from both `channel-research.ts` (`clampFit = (v) => clampScore(v, 60)`) and `twin-reply.ts`. Leave `campaign-eval.ts` and `lib/twin/types.ts`'s private generic `clamp` alone — they're outside this context and `lib/twin/types.ts`'s is already the more general form, not a candidate for merging into an AI-tools-only helper.
- **Gate impact**: Same as finding 2 — editing `_shared.ts` is an unattributed change that forces a full real-model gate run over all 19 tools. If this is done at all, batch it into the same `_shared.ts` commit as finding 2 rather than paying the full-suite cost separately.
