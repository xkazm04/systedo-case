## 1. Ad-experiment ratio math reimplements the shared `metrics/ratios` primitives

- **Severity**: High
- **Category**: duplication
- **File**: `src/lib/ai/experiment-types.ts:39-59`
- **Scenario**: `experiment-types.ts` defines its own divide-guard (`safeRatio`) and its own `variantCtr`/`variantCr`/`variantCpa`/`variantRoas`, even though `src/lib/metrics/ratios.ts:8-29` already exports the identical primitives (`safe`, `ctr`, `cr`, `cpa`, `roas`) with a docstring that says they exist precisely "so both the dashboard analytics ... and the Google-Ads campaign domain ... compute the exact same numbers from one source of truth." `src/lib/campaigns/types.ts:6` already imports from that shared module. The duplication isn't isolated to this file either: `src/lib/images/attribution-types.ts:51-55` independently reinvents the exact same `safeRatio` + `xxxRoas(m) => safeRatio(m.convValue, m.cost)` pattern for creative attribution — three copies of the same four formulas now exist in the codebase.
- **Root cause**: `experiment-types.ts` predates (or was never updated to use) the `metrics/ratios` extraction; its docstring explains it stays framework-free so both client UI and the server store can import it without `firebase-admin`, but `metrics/ratios.ts` is equally framework-free (zero imports) and would have satisfied that constraint just as well.
- **Impact**: Three independent implementations of the same money math (ROAS, CTR, CR, CPA) now have to be kept in sync by hand. They currently agree, but any future fix to the divide-guard (e.g. clamping negative cost, rounding policy) risks being applied to `ratios.ts` while `experiment-types.ts` and `attribution-types.ts` silently keep computing the old numbers — exactly the "two implementations disagree" landmine this scan is watching for, just not triggered yet.
- **Fix sketch**: In `experiment-types.ts`, delete the local `safeRatio` const and rewrite the four `variantXxx` functions as one-line delegations to the shared module, keeping the existing public signatures so no caller changes: `import { ctr, cr, cpa, roas } from "@/lib/metrics/ratios";` then `variantCtr = (m) => ctr(m.clicks, m.impressions)`, `variantCr = (m) => cr(m.conversions, m.clicks)`, `variantCpa = (m) => cpa(m.cost, m.conversions)`, `variantRoas = (m) => roas(m.convValue, m.cost)`. `AdExperiments.tsx` and `pickWinner`/`hasPerformanceBasis` need zero changes. (`attribution-types.ts` is outside this context's file list, so its copy is only cited as corroborating evidence, not part of this fix.)

## 2. `validateLpVariantIdeasRequest` hand-rolls the exact dedupe loop `parseGroundingList` already provides

- **Severity**: High
- **Category**: duplication
- **File**: `src/lib/ai/validation.ts:682-693`
- **Scenario**: Every other "bounded string list" field in this file (`traits`, `always`, `never`, `examples`, `avoid`, `samples`, `localities`, `competitors`, and even `keywords` on `validateChannelResearchRequest`) is parsed through the shared `parseGroundingList(v, maxCount, maxLen)` helper defined at `src/lib/ai/validation.ts:791-805`. `validateLpVariantIdeasRequest`'s `keywords` field instead reimplements the identical trim → cap-length → lowercase-dedupe → cap-count loop inline, byte-for-byte the same logic, just not wired to the helper. The two versions have already drifted in behaviour: `parseGroundingList` scans up to `maxCount * 2` raw items to backfill `maxCount` valid ones (so blanks/duplicates in the input don't starve the result), while the inline copy only ever looks at the first 20 raw items — if a couple of those 20 are blank or duplicate keywords, the LP-variant tool silently returns fewer than 20 keywords even when 20 valid ones existed further down the caller's array.
- **Root cause**: `parseGroundingList` was very likely extracted later (it services the twin-voice, channel-research, and style-training validators) and `validateLpVariantIdeasRequest` — sitting earlier in the file — was never retrofitted to use it.
- **Impact**: Two copies of the same sanitisation logic to maintain, plus a real (if narrow) correctness gap: the LP-variant-ideas tool can under-deliver keywords compared to every sibling tool that uses the shared helper, and a future fix to `parseGroundingList`'s dedupe semantics won't propagate here.
- **Fix sketch**: Replace lines 682-693 with `const keywords = parseGroundingList(o.keywords, 20, 120);` and drop the now-unused local `seen`/loop. `parseGroundingList` is a hoisted function declaration, so no reordering is required. Output shape is unchanged (`string[]`, same caps), so `LpVariantIdeasRequest.keywords` callers are unaffected.

## 3. Seventeen of the eighteen `AiResponse<T>` alias types are never imported anywhere

- **Severity**: Medium
- **Category**: dead-code
- **File**: `src/lib/ai-types.ts:214, 254, 283, 310, 357, 477, 537, 577, 607, 658, 721, 771, 831, 883, 975, 1014, 1065`
- **Scenario**: `ai-types.ts` defines a named `AiResponse<T>` alias for every tool (`BriefResponse`, `AnalysisResponse`, `MonthlyRecapResponse`, `ChatResponse`, `CampaignReportResponse`, `TwinReplyResponse`, `TwinStyleResponse`, `RepurposeResponse`, `LocalReviewReplyResponse`, `ArticleDraftResponse`, `CohortDiagnosisResponse`, `KeywordClustersResponse`, `ComparisonOutlineResponse`, `LpVariantIdeasResponse`, `LeadSourceDiagnosisResponse`, `ChannelResearchResponse`, `OnboardingScanResponse`). A repo-wide grep (`src/` and `tests/`, excluding node_modules) for each of these 17 names finds **zero** references outside their own declaration line. The one sibling that *is* real, `AdResponse` (line 153), is imported once, by `tests/ai-asistent.spec.ts:3`. Route handlers (e.g. `src/app/api/ai/route.ts:85`) instead type their return as `AiResponse<unknown>` / the bare result type directly, confirming the alias convention was never actually adopted by callers.
- **Root cause**: The aliases were added as documentation/scaffolding alongside each tool's `Request`/`Result` pair (following the one real precedent, `AdResponse`), but nothing ever needed the concrete `AiResponse<XResult>` name — callers either use the generic `AiResponse<T>` or just the bare `XResult`.
- **Impact**: Low behavioural risk (pure type-level, zero runtime cost) but real discoverability cost: a reader scanning for "what does this tool return" sees 18 candidate exports and has to check usage to learn that 17 are inert, and every new tool added to this file will likely copy the same now-dead pattern (`ChannelResearchResponse`, `OnboardingScanResponse` — the two newest tools — already did).
- **Fix sketch**: Delete the 17 unused `export type XResponse = AiResponse<XResult>;` lines listed above. Keep `AdResponse` (still referenced by the test fixture). If a future tool genuinely needs a named response type, add it back at that point — YAGNI in the meantime.

## 4. `AD_SEED_LIMITS` and `PIPELINE_KEYWORDS_MAX` are hand-copied mirrors of magic numbers buried in `validation.ts`

- **Severity**: Medium
- **Category**: duplication
- **File**: `src/lib/ai/handoff.ts:11`
- **Scenario**: `handoff.ts:11` defines `AD_SEED_LIMITS = { product: 200, benefits: 600, audience: 300 }`, whose own comment says it exists so a handed-off form "submits without edits" against `validateAdRequest`. Those three numbers are really defined in `src/lib/ai/validation.ts:86-93` as inline literals (`product.length > 200`, `benefits.length > 600`, `audience.length > 300`) — there is no shared constant, just two people who typed the same three numbers in two files. The same shape of duplication exists a second time: `src/lib/ai/pipeline.ts:22` defines `PIPELINE_KEYWORDS_MAX = 60` with a comment saying it mirrors `validateKeywordClustersRequest`, whose actual cap is the inline literal `o.keywords.slice(0, 60)` at `src/lib/ai/validation.ts:767`.
- **Root cause**: `validation.ts`'s own docstring explains it is deliberately kept out of `ai-types.ts` "so the client never pulls in server-only validation logic" — so `handoff.ts` and `pipeline.ts` (both consumed by client components: `ContentBriefGenerator.tsx`, `ContentPipeline.tsx`) correctly avoid importing `validation.ts` directly, but the fix was to re-type the numbers rather than give them one client-safe home.
- **Impact**: If `validateAdRequest`'s benefits cap or `validateKeywordClustersRequest`'s keyword cap ever changes, nothing forces `AD_SEED_LIMITS` or `PIPELINE_KEYWORDS_MAX` to change with it — the seed/pipeline helpers would keep producing values sized for the *old* limit, which the route would then reject or silently truncate differently than intended.
- **Fix sketch**: Move the three `AD_SEED_LIMITS` numbers and the `60` keyword cap into `src/lib/ai-types.ts` as named constants (it already hosts the analogous `AD_LIMITS`/`SEO_LIMITS` pattern for the generated-content limits, e.g. lines 126-131), then have `validation.ts`, `handoff.ts`, and `pipeline.ts` all import the same constants from there.
- **Build risk**: Do not fix this by importing from `validation.ts` into `handoff.ts`/`pipeline.ts` — `validation.ts` is explicitly documented as server-only-adjacent and both target files are transitively imported by client components (`ContentBriefGenerator.tsx`, `ContentPipeline.tsx`); `tsc` would not catch a server-only leak here (per this project's known client/server blind spot), only `next build` would. Routing the shared constants through `ai-types.ts` — which is already framework-free and imported by both sides today — avoids that risk entirely.

## 5. The same "Czech label / English label / picker function" trio is copy-pasted three times in `ai-types.ts`

- **Severity**: Low
- **Category**: structure
- **File**: `src/lib/ai-types.ts:223-237`
- **Scenario**: `ANALYSIS_PERIOD_LABELS` / `ANALYSIS_PERIOD_LABELS_EN` / `analysisPeriodLabel()` (lines 223-237) is structurally identical to `EVAL_PRIORITY_LABELS` / `EVAL_PRIORITY_LABELS_EN` / `evalPriorityLabel()` (lines 316-336) and to `LEAD_SOURCE_CAUSE_LABELS` / `LEAD_SOURCE_CAUSE_LABELS_EN` / `leadSourceCauseLabel()` (lines 894-918): a `Record<T, string>` in Czech, the same `Record<T, string>` in English, and a one-line function that does `locale === "en" ? EN : CS)[key]`. All three pickers are genuinely used by the UI (`analysisPeriodLabel` in `PerformanceAnalyst.tsx`/`MonthlyReport.tsx`/report pages, `evalPriorityLabel` in `ReportView.tsx`, `leadSourceCauseLabel` in `LeadSourceDiagnosisPanel.tsx`), so this is pure structural duplication, not dead code.
- **Root cause**: Each tool's Czech/English label pair was added independently as that tool was built, following the previous tool's pattern by copy-paste rather than factoring out the shared shape.
- **Impact**: ~30 lines of repeated boilerplate; adding a 16th localized enum (this file already has 3 of these plus a bare Czech-only `PLATFORM_LABELS`/`TONE_LABELS`/`CONTENT_TYPE_LABELS`/`COMPARE_INTENT_LABELS`) means copy-pasting the trio a fourth time instead of reusing a generic helper. No behavioural risk — this is a pure readability/consistency nit.
- **Fix sketch**: Add one generic helper in `ai-types.ts`, e.g. `function pickLabel<T extends string>(key: T, locale: SupportedLocale, cs: Record<T, string>, en: Record<T, string>): string { return (locale === "en" ? en : cs)[key]; }`, and rewrite `analysisPeriodLabel`, `evalPriorityLabel`, and `leadSourceCauseLabel` as one-line callers of it. The three `_LABELS`/`_LABELS_EN` Record constants stay as-is (they're the actual data, not the duplication); only the picker function body collapses to a single delegating line each.
